# Exemplar MDX Doc Template

This is the canonical shape for website docs pages under `src/pages/docs/*.mdx`. New or updated pages generated by `apply-product-docs` must match this structure. Derived from the current `src/pages/docs/tasks.mdx`.

## Frontmatter

```yaml
---
layout: ../../layouts/DocsLayout.astro
title: "<Feature Title>"
description: "<One-sentence SEO description, 100-160 chars>"
---
```

**Rules:**
- `layout` is always `../../layouts/DocsLayout.astro` (relative from `src/pages/docs/`).
- `title` is human-readable ("Task Execution", not "tasks").
- `description` describes what the page covers from a user perspective.

## Imports

After the frontmatter and a blank line, import components used in the body:

```mdx
import BrowserFrame from '../../components/ui/BrowserFrame.astro'
```

**Path**: `'../../components/ui/BrowserFrame.astro'` (NOT `'../../components/BrowserFrame.astro'`). The component lives under `components/ui/`.

Additional imports only when needed (e.g., `<Callout>` components).

## Body structure

1. **H1** — matches the frontmatter title.
2. **Intro paragraph** — 1-3 sentences framing what this surface is and why it matters.
3. **Hero screenshot** — `<BrowserFrame>` immediately after intro:
   ```mdx
   <BrowserFrame src="/screenshots/<canonical-filename>.png" alt="<alt from manifest.json description>" />
   ```
4. **Auto-generated body** (bounded by sync markers):
   ```mdx
   {/* SYNC:START */}
   
   ...product markdown body converted to MDX, including additional images and headings...
   
   {/* SYNC:END */}
   ```
5. **Website-only chrome below SYNC:END** — CTAs, crosslinks to `/research/`, related-docs callouts. Never touched by sync.

## Image conventions

- All screenshots live under `/screenshots/` (served from `public/screenshots/`).
- Product markdown uses `/screengrabs/foo.png` — the skill rewrites to `/screenshots/foo.png` during sync.
- Hero image (first image on page) uses `<BrowserFrame src=… alt=…>`.
- Subsequent images use plain markdown `![alt](/screenshots/foo.png)` unless they're meant as visual features (in which case also `<BrowserFrame>`).

## Alt-text source

Alt-text comes from the product's `screengrabs/manifest.json` — each screenshot entry has a `description` field that is the canonical alt-text. This skill does NOT copy alt-text from the product's markdown (which may be context-specific); it normalizes alt-text to the manifest description. Pages that want context-specific alt can hand-edit, which the skill then preserves across syncs (warning the user about divergence, not silently overwriting).

## Why sync markers

The product docs are a running spec that evolves with the product. The website needs to mirror that spec BUT also frame it for marketing readers — CTAs, crosslinks, design-system callouts that the product docs don't care about.

Without sync markers, each sync would either (a) clobber the website-only chrome or (b) require fragile diff-merge logic. Sync markers make the boundary explicit: content inside the markers is auto-generated; content outside is website-owned and preserved forever.

The trade-off: website authors can't edit the body content (inside markers) directly — they have to fix upstream in the product repo. This is intentional: it keeps product and website from drifting, and it routes doc corrections to the right source.
